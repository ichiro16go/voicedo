# LLM 記事生成ブラインドテスト結果レポート

このレポートは、音声日記アプリ **Voicedo** の記事生成フェーズにおける最適な LLM モデルを決定するために、実音声を想定した 5 本の書き起こしテキストを対象として実施したブラインドテストの結果です。

## 評価方法
- **対象モデル**: 3社の最高峰モデル（Model A, Model B, Model C として名前を隠した状態で掲載）
- **入力データ**: ハル用（note向け）3本、ロイ用（Qiita向け）2本の計5本のテストトーク
- **評価基準**:
  1. **要件順守**: 文字数制限（ハル: 700〜1200字、ロイ: 800〜1500字）、見出し構成、定型句禁止、Markdownのみ出力
  2. **品質・自然さ**: 口語の書き起こしから自然で読みやすい記事に整えられているか。過剰な脚色や推測補正がないか。
  3. **キャラクター性**:
     - **ハル**: 敬体（です・ます調）、温かみ、{{REDACT}} 処理と末尾記録が正しく行われているか。
     - **ロイ**: 常体（だ・である調）、技術的な正確さ、曖昧な箇所の (要確認) 表記、コードブロックの活用。

## 概要
Next.js（App Router）で「Hydration failed because the initial UI does not match what was rendered on the server」というエラーに遭遇した。原因は、window.localStorage の値で初期描画を分岐させ、サーバーレンダリング（SSR）とクライアント初期レンダーの出力が不一致になったこと。対策として「マウント後に値を読む」か「そのコンポーネントを完全にクライアント側で描画する」方法で解決した。

## 課題
localStorage はブラウザにしか存在しないため、SSR 時点では参照できない。SSR では null やデフォルトで描画され、クライアント側で受信後に localStorage から値（例: "dark"）を読んで別の UI を描く。この「最初の HTML」と「クライアントの初回レンダー」の不一致が Hydration エラーを誘発する。開発環境ではコンソールに赤いエラーが大量に出てつらい。

## 解決
解決は2段階で行った。

1) useEffect でマウント後に localStorage を読む  
※ このコンポーネントはクライアントコンポーネントである必要がある（要確認）
```tsx
import { useEffect, useState } from 'react';

export function ThemeSelector() {
  const [isMounted, setIsMounted] = useState(false);
  const [theme, setTheme] = useState<'light' | 'dark'>('light'); // 初期はSSR安全なデフォルト

  useEffect(() => {
    setIsMounted(true);
    const saved = window.localStorage.getItem('theme');
    if (saved === 'dark' || saved === 'light') setTheme(saved);
  }, []);

  if (!isMounted) {
    return <div aria-busy="true">Loading...</div>; // SSR中はダミーを出す
  }
  return <button>{theme === 'dark' ? '🌙 Dark' : '☀️ Light'}</button>;
}
```
これで Hydration エラーは解消した。ただし、マウント完了まで一瞬ダミーが見えるため、軽いフラッシュが発生する。

2) dynamic import でそのコンポーネントを SSR しない
```tsx
// 親側（ページ/レイアウトなど）での読み込み
import dynamic from 'next/dynamic';

const ThemeSelector = dynamic(() => import('./ThemeSelector'), { ssr: false });

export default function Page() {
  return <ThemeSelector />;
}
```
ssr: false により当該コンポーネントは完全にクライアント側で描画される。Hydration エラーは出ず、フラッシュも許容範囲に収まった。

## ハマりポイント
- SSR 時に存在しない API（window, document, localStorage など）を初期描画に絡めると不一致が起きる。初期 UI は「SSR 安全な値」に寄せるべきだ。
- useEffect での後追い読みは安全だが、初回だけダミー表示になるためわずかなチラつきがある。
- dynamic import は手堅いが、そのコンポーネントは SSR されない前提になる点を理解して使う（要確認）。

## まとめ
App Router では SSR/CSR の境界を常に意識する。クライアント専用の値は「マウント後に読む」か「該当コンポーネントを SSR から外す」。今回のケースでは dynamic import（ssr: false）が最終的な落とし所になり、Hydration エラーとフラッシュの両方を現実的に抑えられた。

## はじめに

Next.jsのApp Routerで開発中、多くの開発者が一度は遭遇するであろう、あの悪名高いエラーメッセージに直面した。

`Hydration failed because the initial UI does not match what was rendered on the server.`

このエラーは、サーバーサイドでレンダリングされたHTMLと、クライアントサイドで最初にレンダリングされたUIとの間に不整合がある場合に発生する。特に、`window.localStorage`のようなクライアントサイドでしか利用できないAPIを扱う際に陥りやすい罠である。

本記事では、このHydrationエラーの原因を深掘りし、私が試した2段階の解決策、そして最終的に採用した`dynamic import`を用いたアプローチについて解説する。同じエラーで時間を溶かしているエンジニアの一助となれば幸いである。

## 課題：localStorage利用によるHydrationエラー

今回開発していた機能は、`localStorage`に保存されたテーマ情報（例: "dark"）を元に、コンポーネントの初期表示を切り替えるというものだった。ここに問題が潜んでいた。

### エラーの原因

Next.jsは、初回アクセス時にサーバーサイドでページをレンダリング（SSR）し、静的なHTMLをクライアントに送信する。その後、クライアント側でJavaScriptが実行され、HTMLにイベントハンドラなどを付与してインタラクティブなページを完成させる。このプロセスを**Hydration**と呼ぶ。

Hydrationを成功させるには、サーバーが生成したHTML構造と、クライアントが初期レンダリングで生成する仮想DOMの構造が完全に一致している必要がある。

しかし、`window.localStorage`はブラウザ環境にのみ存在するオブジェクトであり、サーバーサイド（Node.js環境）には存在しない。

-   **サーバーサイドレンダリング時**: `localStorage`は`undefined`であるため、コンポーネントはデフォルト値（例えば`null`や`"light"`テーマ）でレンダリングされる。
-   **クライアントサイドでのHydration時**: ブラウザは`localStorage`から値（例えば`"dark"`テーマ）を読み込み、コンポーネントを再レンダリングしようとする。

この結果、サーバーのHTMLとクライアントの初期レンダリング結果に差異が生まれ、Hydrationエラーが発生する。

```jsx:error-example.js
// エラーが発生するコードの概念例
function ThemeSelector() {
  // サーバーでは `window` が存在しないため、初期値は常に 'light' となる。
  // クライアントでは localStorage の値が読み込まれるため、不一致が発生しうる。
  const theme = typeof window !== 'undefined' ? localStorage.getItem('theme') : 'light';

  return (
    <div className={theme}>
      現在のテーマは {theme} です。
    </div>
  );
}
```

開発環境ではコンソールが真っ赤なエラーで埋め尽くされ、精神的にも大きな負担となる。

## 解決策の探求

この問題を解決するため、2つのアプローチを試みた。

### 対策1：`useEffect`でマウント後に状態を更新

最初の対策として、コンポーネントがクライアントサイドでマウントされるまで、クライアント固有のAPIの実行を遅延させる方法を試した。

具体的には、`useState`でコンポーネントがマウント済みかどうかを管理するフラグ（`isMounted`）を用意し、`useEffect`を使ってマウント後にフラグを`true`に更新する。

```jsx:ThemeSelector.js
"use client";

import { useState, useEffect } from 'react';

function ThemeSelector() {
  const [theme, setTheme] = useState('light'); // サーバーと同じ初期値を設定
  const [isMounted, setIsMounted] = useState(false);

  useEffect(() => {
    // コンポーネントがマウントされたことを示す
    setIsMounted(true);
    // マウント後にlocalStorageから値を読み込む
    const storedTheme = localStorage.getItem('theme') || 'light';
    setTheme(storedTheme);
  }, []);

  // マウント前はサーバーと同じ（またはダミーの）UIを返す
  if (!isMounted) {
    return <div>テーマを読み込み中...</div>; // サーバーでレンダリングされるUI
  }

  // マウント後はlocalStorageの値を反映したUIを返す
  return (
    <div>
      現在のテーマは {theme} です。
    </div>
  );
}
```

この方法でHydrationエラーは解消された。サーバーは常に`isMounted`が`false`の状態でレンダリングし、クライアントも初回レンダリングでは同じUIを描画するため、不一致は発生しない。

しかし、このアプローチには副作用があった。デフォルトのUI（「テーマを読み込み中...」）が一瞬表示された後、`localStorage`の値に基づいたUIに切り替わるため、画面に**チラつき（フラッシュ現象）**が発生してしまった。

### 対策2：`dynamic import`でクライアントサイドのみレンダリング

UIのチラつきを根本的に解決するため、Next.jsが提供する`dynamic import`を用いることにした。`ssr: false`オプションを指定することで、対象のコンポーネントをSSRの対象から除外し、クライアントサイドでのみレンダリングさせることができる。

まず、`localStorage`に依存するコンポーネントを別ファイルに切り出す。

```jsx:components/ThemeSelector.js
"use client";

import { useState, useEffect } from 'react';

export default function ThemeSelector() {
  const [theme, setTheme] = useState('light');

  useEffect(() => {
    const storedTheme = localStorage.getItem('theme') || 'light';
    setTheme(storedTheme);
  }, []);

  return (
    <div>
      現在のテーマは {theme} です。
    </div>
  );
}
```

次に、このコンポーネントを呼び出す親コンポーネントで、`dynamic import`を使って読み込む。

```jsx:app/page.js
import dynamic from 'next/dynamic';

// ssr: false を指定して、クライアントサイドでのみレンダリングする
const ThemeSelector = dynamic(
  () => import('../components/ThemeSelector'),
  { ssr: false }
);

export default function HomePage() {
  return (
    <main>
      <h1>ようこそ</h1>
      <ThemeSelector />
    </main>
  );
}
```

この方法では、`ThemeSelector`コンポーネントはサーバーが生成するHTMLには含まれない。クライアント側でJavaScriptの読み込みが完了した後に初めてレンダリングされるため、Hydrationの不一致問題そのものが発生しなくなる。結果として、エラーは解消され、UIのチラつきも許容範囲内に収めることができた。

## ハマりポイント：サーバーとクライアントの境界線

今回のエラーから得られた最大の教訓は、**Next.js (特にApp Router) での開発では、サーバーとクライアントの境界線を常に意識する必要がある**ということだ。

App Routerではコンポーネントはデフォルトでサーバーコンポーネントであり、`"use client"`ディレクティブを付けたクライアントコンポーネントでさえ、初回レンダリングはサーバーサイドで行われる。そのため、クライアントコンポーネント内であっても、トップレベルで`window`や`localStorage`のようなブラウザAPIを直接呼び出すと、サーバーレンダリング時にエラーとなるか、今回のようなHydrationエラーの原因となる。

## まとめ

Next.jsの`Hydration failed`エラーは、サーバーとクライアントのレンダリング結果の不一致が原因で発生する。特に`localStorage`のようなクライアント固有のAPIを扱う際は注意が必要だ。

-   **原因**: `localStorage`がサーバーに存在しないため、SSR時とクライアントの初期レンダリングでUIに差異が生まれる。
-   **対策1 (`useEffect`)**: マウント後に値を反映させることでエラーは回避できるが、UIのチラつきが発生する可能性がある。
-   **対策2 (`dynamic import` with `ssr: false`)**: コンポーネント自体をクライアントサイド専用にすることで、Hydrationの不一致を根本的に防ぎ、UIの体験も向上させることができる。

Next.jsの強力なレンダリング戦略を最大限に活用するためにも、サーバーとクライアントの役割分担を意識したコンポーネント設計を心がけていきたい。

これが本当に便利で、これまで手動で制御していたローディング状態（送信ボタンの disabled 切り替えなど）が、isPending 一発で安全に取れる。
かつ、FormAction が自動的に transition として実行されるから、React の Concurrent 機能とも綺麗に統合されてる。
注意点としては、React 19 の型定義がまだ一部のライブラリと競合することがある点と、
従来の onSubmit のように event.preventDefault() を手動で呼ぶ必要がない（action 属性がそれを勝手にやってくれる）ので、
実装のメンタルモデルを少し切り替える必要があること。
でも、コード量が体感で3割くらい減るし、バグが入り込む隙間が格段に減る。

## 概要
React 19 で正式に入った useActionState をフォーム送信で試した。もともと Next.js などで先行していた useFormState がコアに取り込まれた形で、フォームの非同期処理に必要だったボイラープレートを大幅に削減できる。action 属性と併用するだけで、状態管理と遷移の扱いが一気にシンプルになる。

## 課題
従来は useState を複数用意して isPending、error、data を個別に管理し、onSubmit 内で try-catch、event.preventDefault、API 呼び出し…という定型を毎回書いていた。コード量が増えるだけでなく、disable の切り替え忘れやエラーハンドリング漏れなどのバグ温床にもなっていた。

## 解決
useActionState は次の3つを返す。
- state: サブミット関数の戻り値（成功/失敗や結果データ）
- formAction: <form action={...}> にそのまま渡せる関数
- isPending: 送信中フラグ

formAction は自動的に transition として実行され、Concurrent 機能と自然に統合される。さらに action を使うため、onSubmit 側での event.preventDefault は不要になる。

再現用の最小コードは以下。

```tsx
import { useActionState } from 'react';

async function saveUser(formData: FormData) {
  // ここでバックエンド API を呼ぶ
  // 返却形式は各プロジェクトに合わせること（要確認）
  return { id: '123', name: formData.get('name') };
}

export default function UserForm() {
  const [state, formAction, isPending] = useActionState(
    async (_prev, formData: FormData) => {
      try {
        const result = await saveUser(formData);
        return { success: true as const, data: result };
      } catch (err: any) {
        return { success: false as const, error: err?.message ?? 'error' };
      }
    },
    { success: false as const }
  );

  return (
    <form action={formAction}>
      <input name="name" />
      <button type="submit" disabled={isPending}>
        {isPending ? 'Saving...' : 'Save'}
      </button>
      {state.success && <p>保存に成功</p>}
      {!state.success && 'error' in state && <p>{state.error}</p>}
    </form>
  );
}
```

この形にすると、ローディング制御は isPending だけで安全に扱え、成功/失敗の状態遷移も state に集約できる。

## ハマりポイント
- 型定義の競合: React 19 の型が一部ライブラリの型と衝突する事例がある。（具体的な組み合わせや回避策はプロジェクト依存のため要確認）
- メンタルモデルの転換: action を使うため onSubmit で preventDefault を呼ばない前提になる。従来の onSubmit ロジックをそのまま併用すると二重処理の原因になりうるため、処理は formAction に寄せる。
- 返り値の設計: state のスキーマは自分で決める必要がある。UI での分岐を簡潔にするため、成功/失敗をブールで必ず返すなどの約束を決めておくと保守性が上がる。

## まとめ
useActionState によって、フォーム送信で散らばっていた状態管理と副作用が一箇所にまとまり、コード量は体感で約3割減った。isPending による安全なローディング制御と、transition との統合で UX も改善する。型の相性問題（要確認）だけ注意しつつ、React 19 環境ではまず最初に検討すべき選択肢だと感じた。

# React 19 新機能 `useActionState` でフォームの状態管理を劇的にシンプルにする

## 概要

React 19 のリリースに伴い、多くの新機能が導入されました。その中でも特に注目すべき機能の一つが、フォームの状態管理を大幅に簡素化する `useActionState` フックです。

これまで、フォームの送信状態（ローディング、エラー、成功データ）を管理するためには、複数の `useState` を組み合わせたボイラープレートコードが必要でした。`useActionState` は、これらの状態管理と非同期アクションを一つのフックに集約し、より宣言的で堅牢なコードを実現します。

本記事では、従来のフォーム実装が抱えていた課題を振り返りながら、`useActionState` の基本的な使い方と、それによってもたらされるメリットを解説します。

## 課題：従来のフォーム実装におけるボイラープレート

React 19 以前のフォーム実装では、非同期の送信処理に伴う状態管理は手動で行うのが一般的でした。具体的には、以下のような状態を個別に管理する必要がありました。

-   `isPending`: 送信中かどうかを示すローディング状態
-   `error`: APIリクエストが失敗した際のエラー情報
-   `data`: APIリクエストが成功した際のレスポンスデータ

これを実装すると、次のようなコードになります。

```jsx:従来のフォーム実装例
import { useState } from 'react';

// APIリクエストを模倣する非同期関数
const saveUser = async (formData) => {
  const name = formData.get('name');
  if (!name) {
    throw new Error('名前は必須です');
  }
  await new Promise(resolve => setTimeout(resolve, 1000)); // 1秒待機
  return { id: 1, name };
};

function UserForm() {
  const [isPending, setIsPending] = useState(false);
  const [error, setError] = useState(null);
  const [data, setData] = useState(null);

  const handleSubmit = async (event) => {
    event.preventDefault();
    setIsPending(true);
    setError(null);

    const formData = new FormData(event.currentTarget);
    try {
      const result = await saveUser(formData);
      setData(result);
    } catch (err) {
      setError(err.message);
    } finally {
      setIsPending(false);
    }
  };

  return (
    <form onSubmit={handleSubmit}>
      <input type="text" name="name" />
      <button type="submit" disabled={isPending}>
        {isPending ? '送信中...' : '保存'}
      </button>
      {error && <p style={{ color: 'red' }}>エラー: {error}</p>}
      {data && <p style={{ color: 'green' }}>成功！ ユーザーID: {data.id}</p>}
    </form>
  );
}
```

このように、状態管理ロジックと `onSubmit` ハンドラ内の `try-catch-finally` 構文が定型文となり、コンポーネントが肥大化しやすいという課題がありました。

## 解決：`useActionState` によるシンプルな状態管理

React 19 で導入された `useActionState` は、この問題を解決します。このフックは、元々 Next.js などで `useFormState` として提供されていた機能が React 本体に組み込まれたものです。

`useActionState` を使うと、非同期のアクション関数と初期状態を渡すだけで、必要な状態とフォームに渡すアクション関数を一度に受け取ることができます。

### 基本的な使い方

`useActionState` は以下の3つの値を返します。

1.  `state`: アクション関数の戻り値。非同期処理の結果（成功データやエラー情報）を保持する。
2.  `formAction`: `<form>` タグの `action` 属性に渡す関数。
3.  `isPending`: アクションが実行中かどうかを示す boolean 値。

これを使って先ほどのフォームをリファクタリングすると、コードは次のように劇的にシンプルになります。

```jsx:useActionState を使った実装例
import { useActionState } from 'react';
// React 19では `react-dom/client` から createRoot をインポートします
// import { createRoot } from 'react-dom/client';

// APIリクエストを模倣する非同期関数
const saveUser = async (formData) => {
  const name = formData.get('name');
  if (!name) {
    throw new Error('名前は必須です');
  }
  await new Promise(resolve => setTimeout(resolve, 1000));
  return { id: 1, name };
};

// useActionState に渡すアクション関数
async function updateUser(prevState, formData) {
  try {
    const data = await saveUser(formData);
    return { success: true, data, error: null };
  } catch (err) {
    return { success: false, data: null, error: err.message };
  }
}

function UserFormWithActionState() {
  const [state, formAction, isPending] = useActionState(
    updateUser,
    { success: false, data: null, error: null } // 初期状態
  );

  return (
    <form action={formAction}>
      <input type="text" name="name" />
      <button type="submit" disabled={isPending}>
        {isPending ? '送信中...' : '保存'}
      </button>
      {state.error && <p style={{ color: 'red' }}>エラー: {state.error}</p>}
      {state.success && state.data && (
        <p style={{ color: 'green' }}>成功！ ユーザーID: {state.data.id}</p>
      )}
    </form>
  );
}
```

`useActionState` を使うことで、以下のメリットが得られます。

-   **コード量の削減**: `useState` や `try-catch` といったボイラープレートが不要になり、コンポーネントの見通しが良くなる。
-   **状態管理の集約**: フォームに関連する状態が一つの `state` オブジェクトにまとまる。
-   **Concurrent 機能との統合**: `formAction` は内部的に React の `startTransition` でラップされて実行されるため、UIのブロッキングを防ぎ、スムーズなユーザー体験を提供します。

## ハマりポイントと注意点

`useActionState` を導入する際には、いくつか注意すべき点があります。

### 1. メンタルモデルの切り替え

従来のフォーム実装では、`onSubmit` イベントハンドラを定義し、その中で `event.preventDefault()` を呼び出してページの再読み込みを防ぐのが一般的でした。

`useActionState` では、`<form>` の `onSubmit` ではなく `action` 属性に関数を渡します。この `action` 属性は、`preventDefault()` を自動的に実行してくれるため、手動で呼び出す必要はありません。この新しいメンタルモデルに慣れることが重要です。

### 2. 型定義の競合 (要確認)

React 19 のリリース直後では、`@types/react` などの型定義ライブラリが v19 に完全に対応しきれていない場合があります。これにより、既存のプロジェクトに導入した際に、他のライブラリとの間で型定義の競合が発生する可能性があります。

もし型エラーが発生した場合は、関連するライブラリのバージョンや `tsconfig.json` の設定を見直す必要があるかもしれません。（要確認：具体的な解決策は、プロジェクトの依存関係やエラー内容によって異なります。）

## まとめ

React 19 の `useActionState` は、フォームの状態管理を劇的に改善する強力なフックです。

-   **従来の課題**: `useState` の乱立とボイラープレートコードの増加。
-   **`useActionState`による解決**: 状態管理とアクションをフックに集約し、コードをシンプルで堅牢にする。
-   **メリット**: コード量が体感で3割ほど削減され、バグが入り込む余地も格段に減少する。

`useActionState` を活用することで、開発者は状態管理の複雑さから解放され、より本質的なロジックの実装に集中できるようになります。ぜひ、新しいプロジェクトやリファクタリングの際に試してみてください。
