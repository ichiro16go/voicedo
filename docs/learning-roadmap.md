Voicedoプロジェクトを教材にして、マーケティング・PM・分析スキルを身につけるための学習ロードマップ。

## Part 1：この壁打ちで実際に使った思考フレーム（メタ分析）

### 議論の進め方
```
問題を聞く → 仮説を1つ立てる → 検証質問を投げる →
フレームで整理 → 数字で裏付け → 反対視点で叩く → 結論
```
これは**仮説検証思考（Hypothesis-Driven Thinking）**。コンサル/PMの基本動作。

### 使ったフレームワーク一覧

| 場面 | 使ったフレーム | 名前 |
|---|---|---|
| ペルソナ設計 | 痛点・代替手段・支払い意欲で分解 | **JTBD（Jobs To Be Done）** |
| 競合分析 | ポジショニング2軸マップ | **Positioning Map** |
| 市場機会発見 | 空白象限を狙う | **Blue Ocean Strategy** |
| 差別化 | 強み/弱み×内部/外部 | **SWOT** |
| 価格設計 | LTV÷CAC, 粗利率 | **Unit Economics** |
| 撤退ライン | 事前に失敗条件を決める | **Pre-Mortem** |
| MVP | 仮説検証段階で機能を削る | **Lean Startup（BML）** |
| 継続率 | D1/D7/D30コホート | **Retention Cohort** |
| グロース | ループ設計、バイラル係数 | **Growth Loops** |
| Free→有料 | フリーミアム心理学 | **Hook Model** |

→ これらは全て1冊1テーマで本がある領域。順に潰せば武器化できる。

## Part 3：思考の「型」を体に染み込ませる日次/週次練習

### 🌅 毎日（5-15分）

#### ① プロダクト分解日記
気になったアプリを毎日1つ、3行で分解：
```
プロダクト名: ●●●
JTBD: 〜〜を〜〜したい人が、〜〜したくて雇う
差別化: 〜〜であること（vs ○○）
```

#### ② 数字で語る習慣
「だいたい」を「具体的に」に変える＝**フェルミ推定**を癖にする。

### 📅 毎週（1-2時間）

#### ③ 競合観察ジャーナル
毎週月曜、競合5本を5分ずつ観察。新機能、LP変更、価格、レビュー傾向。

#### ④ "Lenny's Podcast" 1本聴く
通勤・散歩で。

### 📊 毎月（半日）

#### ⑤ KPIレビュー会（自分1人）
- 仮説通り動いた数字は？
- 想定外だった数字は？ なぜ？
- 次月の打ち手は？

#### ⑥ Pre-mortem 自己診断
「Voicedoが半年後失敗していたとしたら、原因は何か？」を月1回書く。

## Part 4：エンジニア出身ならではの "ズル" できる勝ち筋

エンジニアは数学的思考とツール構築力がある＝マーケでも非エンジニアより速く強くなれる。

### 強みを活かす5つの方法
1. **自分専用ダッシュボードを作る**（PostHog API → Notion 自動配信）
2. **AIで分析を10倍速にする**（ユーザーレビュー1000件をClaude/GPTでカテゴリ分け）
3. **A/Bテストの自前実装**（外部ツール不要）
4. **コード=コミュニケーション媒体に**（技術ブログでエンジニア層信用獲得）
5. **MVP実装速度がそのままディスカバリー速度**（思いつき→3日後にA/Bテスト）

## Part 6：Voicedoを "教材" として最大活用するコツ

### ① 毎週「学んだこと」をnoteに書く
- 「Voicedoの開発でユーザーインタビューを5人やった結果」
- 「muute分析から見えた競合観察の方法」
- 「Free枠を3パターン試してA/Bした結果」
→ 学び×アウトプット二重利益、記事自体がVoicedo宣伝に

### ② Decisionログを残す
重要意思決定を残す：
```
[2026-06-22] Free枠を「30日無制限ハイブリッド」に決定
理由: muuteのType B（課金しなかった組）を取るため
代替案: 月3セッション無料 → 棄却（習慣化に届かない）
結果（3ヶ月後追記）: ...
```
→ 意思決定の質を後から評価できる＝上達の燃料

### ③ 数字を最初から取る
Stage 1の時点でPostHogを入れる。「PMFしてから」では遅い。

### ④ 失敗を全公開する
Build in Public は成功より失敗を出すほうが学びも信頼も大きい。
